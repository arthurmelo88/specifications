openEHR Specifications Release 1.0.2
Release Notes

Overview

Release 1.0.2 of the openEHR specifications includes 15 effective Change Requests (CRs) from a total of 19 (see History), and 3 resolved Problem Reports (PRs). The CRs have the following breakdown:
  • Correct typographical errors: SPEC-249, 255, 257, 258, 274, 284
  • Descriptive text improvements: SPEC-253, 275
  • Minor errors in formal specifications: SPEC-256,  260, 268, 271, 283
  • Minor formal enhancements/clarifications: SPEC-251, 261
  • Rejected/superseded: SPEC-250, 252, 269, 272

Impact

Software implementations:

  • ADL parsers:
    • SPEC-251. Parsers other than the EIffel reference parser and the java reference project parser will need to be updated to take account of the added syntax for pattern/interval duration constraints.
    • Due to SPEC-268, parsers need to be upgraded to handle parentheses around type names in embedded dADL blocks in ADL archetypes, such as the C_DV_QUANTITY blocks found in many archetypes. It is recommended that tools handle both forms for the time being, with conversion to the legal form being made in environments where it is safe to do so.See wiki page on tooling issues.
  • Rerence Model implementations: 
    • the regular expression pattern for legal ARCHETYPE_IDs has been corrected. See SPEC-260 and the Support IM specification.
    • some implementations may have used the US spelling for the ELEMENT.null_flavour attribute. This has been corrected to international English spelling. See SPEC-283.
    • Implementations of the DV_QUANTIFIED types will need to be augmented to include the improved semantics for accuracy. See SPEC-261.

XML-schemas

Existing Archetypes and Templates

  • The identifiers and filenames of some existing archetypes violate the rules published in the Support IM specification, due to the use of the ".v1draft" pattern. For the moment, tools should continue to support the old form, converting to the new form where practical. See the wiki page on this issue. A new draft of the rules for archetype and template identification and referencing will soon be published.
  • Due to the correction of SPEC-284, a few existing ADL archetypes only have misnamed subsections in the ontology section: term_binding should be term_bindings; constraint_binding should be constraint_bindings (i.e. both in the plural). Tools should accept both spellings for some time and automatically convert to the new spelling where safe to do so on saving.
  • Due to SPEC-268, many existing archetypes contain an illegal form of dADL block, introduced by a type name (most often C_DV_QUANTITY) that should be enclosed in parentheses but is not. Tools should perform conversions in environments where it is safe to do so. See wiki page on tooling issues.
