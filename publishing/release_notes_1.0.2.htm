openEHR Specifications Release 1.0.2
Release Notes

Overview

Release 1.0.2 of the openEHR specifications includes 15 executed Change Requests (CRs) from a total of 19 proposed (see History), and 3 resolved Problem Reports (PRs). The CRs have the following breakdown:
  • Correct typographical errors: SPEC-249, 255, 257, 258, 274, 284
  • Descriptive text improvements: SPEC-253, 275
  • Minor errors in formal specifications: SPEC-256,  260, 268, 271, 283
  • Minor formal enhancements/clarifications: SPEC-251, 261
  • Rejected/superseded: SPEC-250, 252, 269, 272
The CRs causing changes to the formal specifications are summarised as follows:

SPEC-251 Allow both pattern and interval constraint on Duration in Archetypes
Affects: ADL, AOM specifications
ADL now supports a constraint on durations allowing both ranges and unit types (weeks, days etc) to be expressed together, e.g.: some_duration matches {PWD/|P0W..P50W|}
This is typically used for pregnancy duration constraints.
SPEC-256 Correct extension_validity in UID_BASED_ID class
Affects: Support IM specification
The text for the has_extension function is changed to indicate that the Result is True if extension is not empty (rather than not Void).
SPEC-260 Correct the regex published for the ARCHETYPE_ID type
Affects: Support IM specification
The proposed solution is as below, and has the following features:
- the characters #, ( and ) are not allowed
- no part (i.e. any section between '-' or '.' delimiters) can be only 1 character long
- no name part can start with a digit
- allows upper case in the alphabetic parts of the id
- limiting the version part of the id to numbers (i.e. no letters)
- not allowing a leading '0' in the version identifier
SPEC-261 Indicate how accuracy is treated over add/subtract operations in DV_QUANTIFIED types
Affects: Data Types IM specification
1. in the class DV_AMOUNT, accuracy_unknown: Boolean is added as a function, and a constant called Unknown_accuracy_value of type Integer, with value = -1 is added. The documentation is changed to indicate that a value of -1, i.e. accuracy = Unknown_accuracy means no accuracy recorded, and the accuracy_unknown function returns True.
2. The class DV_ABSOLUTE_QUANTITY also has accuracy, but defined as a reference value. The accuracy_unknown function is also implemented in this class, but returns True if accuracy = Void (i.e. null).
SPEC-268 Correct missing parentheses in dADL type identifiers
Affects: ADL specification
The dADL grammar is corrected to allow parentheses around type names, as in the following example from the ADL specification:
destinations = <
    ["seville"] = (TOURIST_DESTINATION) <
    ....
>
SPEC-271 Correct minor inconsistencies in ITEM_TABLE specification and example
Affects: Data Structures IM
The ITEM_TABLE.ith_row() function is updated to the following:
ith_row (i:Integer): CLUSTER
require
  i >= 1 and i <= row_count
SPEC-283 Correct spelling of ELEMENT.null_flavor attribute to null_flavour
Affects: Data Structures IM
The null_flavor attribute in the reference model class ELEMENT is renamed to null_flavour to match the published XML-schema.

Impact Analysis

Software implementations:

  • ADL parsers:
    • SPEC-251. Parsers other than the EIffel reference parser and the java reference project parser will need to be updated to take account of the added syntax for pattern/interval duration constraints.
    • Due to SPEC-268, parsers need to be upgraded to handle parentheses around type names in embedded dADL blocks in ADL archetypes, such as the C_DV_QUANTITY blocks found in many archetypes. It is recommended that tools handle both forms for the time being, with conversion to the legal form being made in environments where it is safe to do so.See wiki page on tooling issues.
  • Rerence Model implementations: 
    • the regular expression pattern for legal ARCHETYPE_IDs has been corrected. See SPEC-260 and the Support IM specification.
    • some implementations may have used the US spelling for the ELEMENT.null_flavour attribute. This has been corrected to international English spelling. See SPEC-283.
    • Implementations of the DV_QUANTIFIED types will need to be augmented to include the improved semantics for accuracy. See SPEC-261.

XML-schemas

  • BaseTypes.xsd
    • technical correction to make UID_BASED_ID an abstract class; this has no effect on any system or data.
    • DV_AMOUNT class, a default value is added to accuracy element. The default value will be used when the accuracy is empty. However, when accuracy is not provided, then there will no default value for accuracy.
  • Resource.xsd
    • make AUTHORED_RESOURCE abstract
    • make RESOURCE_DESCRIPTION.details element mandatory, i.e. change minOccurs="0" to minOccurs="1"

Existing Archetypes and Templates

  • The identifiers and filenames of some existing archetypes violate the rules published in the Support IM specification, due to the use of the ".v1draft" pattern. For the moment, tools should continue to support this form, converting to the new form where practical. See the wiki page on this issue. A new draft of the rules for archetype and template identification and referencing will soon be published.
  • Due to the correction of SPEC-284, a few existing ADL archetypes only have misnamed subsections in the ontology section: term_binding should be term_bindings; constraint_binding should be constraint_bindings (i.e. both in the plural). Tools should accept both spellings for some time and automatically convert to the new spelling where safe to do so on saving.
  • Due to SPEC-268, many existing archetypes contain an illegal form of dADL block, introduced by a type name (most often C_DV_QUANTITY) that should be enclosed in parentheses but is not. Tools should perform conversions in environments where it is safe to do so. See wiki page on tooling issues.
